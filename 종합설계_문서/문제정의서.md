## 1. 연구 개발의 필요성

### 1.1 연구 배경

- **이기종 증거 데이터의 폭증:** 클라우드, 엔드포인트(EDR), 네트워크 등 IT 인프라가 다변화되면서 침해 사고 조사에 필요한 아티팩트(Artifact)의 종류와 양이 기하급수적으로 증가하고 있다 [1][2].
- **수동적 분석의 한계:** 분석관이 개별 데이터 소스에 맞춰 각기 다른 포렌식 도구를 수동으로 구동하고, 그 결과를 육안으로 교차 검증하는 노동 집약적 방식으로 인해 글로벌 평균 3~4주의 Backlog가 누적되어 침해사고 대응의 골든타임 확보가 어렵다 [3][5].
- **지능형 오케스트레이션(Orchestration)의 필요성:** 포렌식 업무는 증거 수집·파싱·타임라인 생성 등 반복적이나, 공격 벡터·시스템 환경에 따라 분석 맥락이 매번 달라지는 비정형적 특성을 가지므로 [6], 이 가변성에 동적으로 대응할 수 있는 시스템 수준의 지능형 기술이 필수적이다.

### 1.2 연구 동향

- **Agentic AI의 진화:** 대형 언어 모델(LLM)의 활용이 외부 API를 호출하고 능동적으로 시스템과 상호작용하는 에이전트 기술로 성숙하고 있다 [7][8].
- **MCP(Model Context Protocol)의 부상:** 수많은 외부 도구를 LLM과 규격화된 방식으로 연결하는 MCP가 등장하여 [9], 이기종 도구 통합 제어의 기술적 기반이 마련되었다.
- **Human-in-the-Loop (HITL) 협업 모델 대두:** AI의 블랙박스 추론과 환각(Hallucination)을 통제하기 위해 인간의 직관과 AI를 결합하는 HITL 구조가 필수적인 접근법으로 평가받고 있다 [10][11].
- **디지털 포렌식에 LLM 도입 연구 증가:** 포렌식 분야에서 LLM 도입 연구가 활발히 수행되고 있으나 각각 명확한 한계를 보인다. ForensicLLM은 자연어 기반 분석을 시도하였으나 외부 포렌식 도구와의 연동 인터페이스가 부재하고 [14], CyberSleuth는 웹 공격 포렌식에 특화되어 범용 침해사고 조사에 적용하기 어렵다 [15]. ProvSEEK는 출처 추적에 한정되며 [16], AutoGen DF는 멀티에이전트 구조를 제안하였으나 실제 포렌식 도구 통합 없이 시뮬레이션 수준에 머문다 [17].

### 1.3 문제 정의 및 해결 방안

**문제 1 (비정형적 분석 맥락)**
DFIR 업무는 반복적이나, 공격 벡터·시스템 환경에 따라 분석 흐름이 매번 달라지는 비정형적(Non-uniform) 특성을 가진다. 일률적인 스크립트/룰 기반 자동화로는 이 가변성에 완벽히 대응할 수 없다.

> **해결 1 (LLM 기반 동적 분석 계획):** LLM의 추론 능력을 활용하여 사건 맥락에 맞는 분석 전략을 동적으로 수립하고, 분석관이 자연어로 상황을 입력하면 AI가 최적의 분석 흐름을 자동으로 계획(Dynamic Planning)

**문제 2 (도구별 진입 장벽)**
기존 연구에서 LLM을 도입하고 있으나, 분석관이 여전히 복잡한 쿼리나 도구별 API를 직접 다뤄야 하여 편의성 측면의 진입 장벽이 존재한다. 이기종 포렌식 도구들을 하나의 일관된 맥락(Context) 안에서 제어할 수 있는 통합 인터페이스가 부재한다.

> **해결 2 (MCP 기반 오케스트레이션):** 파편화된 도구들을 MCP 기반으로 래핑(Wrapping)하여, 에이전트가 단일 인터페이스 환경에서 자율적으로 도구를 호출·실행할 수 있는 환경을 구축

**문제 3 (분석 결과의 추적 가능성 부족)**
기존 LLM 기반 시스템은 추론 근거와 원본 증거 간의 역참조(back-reference) 구조가 부재하여, 분석 결과의 검증 가능성과 보고서의 신뢰성이 불충분하다.

> **해결 3 (다단계 HITL 프레임워크 도입):** 전략 수립부터 도구 실행까지의 과정을 7단계로 세분화하고, 매 의사결정 노드마다 분석관이 개입하는 상호작용 프로세스를 통해 추론과 실행의 간극을 해소한다.

### 1.4 문제 해결의 필요성

- **기존 연구의 공백:** 포렌식 LLM 연구들은 도구 연동 부재, 특정 도메인 한정, 범용성 부족, 확장성 제한이라는 공통적 한계를 보이며, MCP 기반 도구 통합 제어와 HITL 협업 구조를 결합한 포렌식 오케스트레이션 에이전트 연구는 아직 초기 단계에 머물러 있어 실용적 수준의 구현 사례가 부족하다.
- **실무적 시급성:** 침해사고 건수는 매년 증가하는 반면 숙련된 포렌식 인력의 공급은 만성적으로 부족하여, 세 가지 문제(비정형적 분석 맥락, 도구 파편화, 추적 가능성 부족)를 기술적으로 보완할 수 있는 본 연구의 수행이 필요하다.

## 2. 연구 개발의 목표 및 내용

### 2.1 연구 개발 목표

이기종 포렌식 도구의 통합 제어와 분석관-AI 간 유기적 협업을 지원하는 **MCP 및 HITL 기반 지능형 오케스트레이션 에이전트 프로토타입** 개발

### 2.2 주요 연구 내용

**[Goal 1] LLM 기반 동적 분석 계획**

- 증거물 입력 및 메타데이터 인식 → 과거 성공 사례 기반 최적 도구 자동 매핑
- 개별 도구 산출물의 의미론적 융합으로 초기 타임라인 자동 구성 및 후속 조사 방향 선제적 제안

**[Goal 2] MCP 기반 오케스트레이션**

- 주요 오픈소스 포렌식 도구의 핵심 기능을 MCP 표준 API로 래핑(Wrapping)
- Agentic AI가 도구별 기능 명세(Tool Description)를 기반으로 자율 호출할 수 있는 환경 구축

**[Goal 3] 다단계 HITL 프레임워크 도입**

- 7단계 순환 루프 구현:
  1. 초기 입력
  2. 전략 도출 → Human 검토
  3. 계획 도출 → Human 검토
  4. Task 할당 → Human 검토
  5. 도구 자동 실행 (MCP)
  6. 결과 분석 → Human 검토
  7. 순환 및 완료

### 2.3 기존 방식 대비 차별성

| 비교 항목      | AS-IS: 현행 포렌식 조사 방식                                                                                                | TO-BE: 제안 플랫폼                                                                                                |
| -------------- | --------------------------------------------------------------------------------------------------------------------------- | ----------------------------------------------------------------------------------------------------------------- |
| 도구 운용 방식 | 분석관이 11개 이상의 도구를 개별 실행하고, 각 도구의 산출물을 수동으로 변환·재입력 [21]                                     | MCP 기반 1:N 단일 프로토콜로 이기종 도구를 통합 제어, 노드 간 데이터 자동 전달                                    |
| AI 활용 가능성 | 기존 포렌식 도구(EnCase, FTK, Volatility 등)는 독립형 설계로, AI와의 연동 인터페이스가 부재하여 LLM 기반 자동화 적용이 곤란 | MCP 서버 래퍼를 통해 기존 도구를 AI 에이전트가 호출 가능한 형태로 표준화, 추가 글루 코드 없이 기능 명세 자동 발견 |

## 3. 연구 개발의 추진전략 및 방법

### 3.1 기술적 구현 방법론

- **MCP 서버의 도구 명세 자동 발견(Auto-Discovery) 구조 설계:** 에이전트가 런타임에 등록된 MCP 서버의 기능 명세(Tool Description)를 동적으로 조회하여, 사전 하드코딩 없이 최적 도구를 선택·호출할 수 있는 환경을 구현한다.
- **노드별 승인 게이트(Approval Gate) 메커니즘 구현:** 7단계 HITL 루프의 각 의사결정 노드에 분석관이 승인·수정·거부를 선택할 수 있는 게이트를 삽입하여, AI 추론과 도구 실행 사이의 무검증 자동 진행을 원천 차단한다.

### 3.2 문제 해결 지향적 기술 검증

- **LLM 동적 분석 계획의 유효성 (문제 1 해결 검증):** 공격 경로·핵심 아티팩트·타임라인이 사전 확정된 통제 시나리오(Ground-Truth) 이미지를 구성하고, LLM이 생성한 분석 계획의 아티팩트 커버리지를 정답지 대비 측정
- **MCP 래퍼의 도구 통합 제어 안정성 (문제 2 해결 검증):** 이기종 도구 2종 이상을 MCP 래퍼로 구동하여 정상 호출·반환 여부 및 원본 증거 무결성(해시값 일치) 확인
- **HITL 프레임워크의 추론-실행 간극 해소 효과 (문제 3 해결 검증):** 7단계 HITL 루프를 적용한 E2E 시나리오에서, 분석관의 검토 개입 없이 자동 실행만 수행한 경우와 비교하여 최종 보고서의 정확성·완전성을 평가

### 3.3 애자일(Agile) 방식의 프로젝트 진행 전략

프롬프트의 환각 현상이나 MCP 연동 시 발생하는 예기치 못한 API 충돌 등 AI 연구 특유의 불확실성을 최소화하기 위해, 짧은 주기로 핵심 기능을 구현하고 피드백을 수용하는 **애자일 스쿼드(Agile Squad)** 방법론을 프로젝트 전반에 채택

## 4. 연구 개발 팀의 구성 및 프로젝트 추진 일정

### 4.1 팀 구성 및 역할 분담

3.3의 애자일 추진 전략에 따라, 본 연구 팀은 직무를 고정하지 않고 Task 단위로 유기적으로 결합하는 **4인 스쿼드** 형태로 운영한다. 특히 매 스프린트의 목표에 맞춰 최적의 인원에게 리더십을 부여하는 **순환형 리더십(Rotating Leadership)** 체제를 도입한다.
